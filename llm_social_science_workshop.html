   # Send a system prompt + user text to a GPT model and return the string response.

   # Parameters
   # ----------
   # prompt      : str   — system-role instructions (your annotation schema / task definition)
   # text        : str   — user-role content (the document/text to be processed)
   # model       : str   — OpenAI model identifier
   # temperature : float — sampling temperature; lower = more deterministic output.
 

def ask_gpt(prompt, text, model="gpt-4o-mini", temperature=0.1):
    """Send a system prompt + user text to GPT and return the response."""
    kwargs = {
        "model": model,
        ## system = your instructions, user = the data to process
        "messages": [
            {"role": "system", "content": prompt},
            {"role": "user", "content": text}
        ],
    }
    ## some models only support default temperature
    if temperature is not None:
        kwargs["temperature"] = temperature
    try:
        response = client.chat.completions.create(**kwargs) #with or without dic
    except Exception:
        ## retry without temperature if the model doesn't support it
        kwargs.pop("temperature", None)
        response = client.chat.completions.create(**kwargs)
    return response.choices[0].message.content
def parse_json_response(raw):
    """LLMs sometimes wrap JSON in code blocks. Strip that and parse."""
    # Remove leading/trailing whitespace from the raw model output.
    # This handles newlines or spaces that often appear before/after code blocks.
    clean = raw.strip()
    if clean.startswith("```"):
        ## strip the ```json ... ``` wrapper that GPT likes to add
            # GPT often returns JSON wrapped in a markdown code fence, e.g.:
        #   ```json
        #   {"label": "positive", "confidence": 0.9}
        #   ```
        # We need to strip that wrapper before parsing.

        # Split on the first newline only, discarding the opening fence line
        # (e.g. "```json" or "```").
        # [1] takes everything AFTER that first line.
        clean = clean.split("\n", 1)[1]
        # Strip the closing ``` fence from the end.
        # rsplit from the right ensures we only remove the last occurrence,
        # in case the JSON content itself contains backticks.
        clean = clean.rsplit("```", 1)[0]
        #  # Parse the cleaned string into a Python dict/list.
    return json.loads(clean)

Part 1: What Are LLMs and How Do We Use Them?

An LLM is basically a neural network trained on a huge amount of text. It learns to predict what comes next in a sequence, and it turns out that’s useful for all sorts of tasks:

  • Classification,
  • Extraction,
  • Translation,
  • Summarization.

We talk to it through an API: send a prompt and some text, get a structured response back.

Important caveat

LLMs generate text. They can (and do) hallucinate: producing confident output that’s just wrong. This is why we pair every application with a battery of verification steps.

Prompt engineering basics

When you call an LLM through the API, you send messages with roles:

  • System message — your instructions (“classify this text into categories X, Y, Z…”)
  • User message — the actual data you want processed
  • Temperature — controls how random or varied the model’s output is. We keep it low (0.1) for consistency.
  • Structured output — ask for JSON so you can parse the result in code

Prompt sensitivity — this matters more than you’d think

Trivial changes to how you word a prompt can flip the output. Capitalizing the options, adding bold formatting, telling the model what not to do can all make changes to the output.

If your classification changes because you bolded a word, that’s a problem. Let’s see how bad this actually is.

  • Testing small prompt variations is a good way to assess prompt sensitivity or robustness.
  • Comparing against a hand-labeled sample is important if you want to know whether the prompt is not just stable, but also accurate
  • Keeping a hold-out set that you never use during prompt iteration is also good practice. Analogy to train/test logic in supervised ML.

Zero-shot, few-shot, fine-tuning

  • Zero-shot — just instructions, no examples. Works surprisingly well for many tasks.
  • Few-shot — include some worked examples in the prompt. Helps with ambiguous cases.
  • Fine-tuning — train the model on your own labeled data. Best results, most effort.

Zero-shot classification

  • We write a prompt listing our four categories with example topics so the LLM knows what we mean
  • We loop over all 20 test texts, send each to GPT, and collect the predictions
## list each category with example topics so the LLM knows what we mean
zero_shot_prompt = """Classify this political text into exactly one category:
- economy (taxes, trade, growth, employment, fiscal policy)
- security (defense, military, terrorism, borders, intelligence)
- social_policy (healthcare, education, welfare, housing, inequality)
- environment (climate, energy, emissions, biodiversity, pollution)

Return ONLY the category name, nothing else."""

Part 3: Named Entity Recognition — spaCy vs LLM

Now for some other things LLMs can do. NER finds people, places, and organizations in text. spaCy does this fast and free. However, it labels “Sweden” and “Geneva” both as GPE (geopolitical entity). It can’t tell you that Sweden is where the person is from and Geneva is where the event happened.

Why? spaCy uses statistical patterns. It labels entity types but can’t reason about context. LLMs read the whole sentence and understand roles.

For example:
sentence: "Senator Smith criticized President Macron for France's energy policy"

entities:
  Smith:  { type: PERSON, role: SPEAKER/CRITIC }
  Macron: { type: PERSON, role: TARGET }
  France: { type: GPE,    role: AFFILIATED_COUNTRY }

Let’s measure against the known transcription.

Validating LLM Transcription

We compare each model’s output against a human ground truth transcript.

  • Precision — are the extracted words real?
  • Recall — were all real words recovered?
  • F1 — overall balance between the two

We ask: did the right words appear?

This is a very simple way of validating this output: there are many more.

def word_level_metrics(extracted, ground_truth):
    # --- preprocessing: lowercase + tokenize into unique word sets ---
    # NOTE: using sets loses word frequency — "the the the" counts same as "the"
    truth_words = set(ground_truth.lower().split())
    extracted_words = set(extracted.lower().split())

    # guard: if ground truth is empty, metrics are undefined
    if not truth_words:
        return {"precision": 0, "recall": 0, "f1": 0}

    # true positives: words present in both outputs
    overlap = truth_words & extracted_words

    # precision: of all words the LLM returned, how many were real?
    # low precision → model is hallucinating words not in the original
    precision = len(overlap) / len(extracted_words) if extracted_words else 0

    # recall: of all real words, how many did the LLM recover?
    # low recall → model is dropping words (under-extraction)
    recall = len(overlap) / len(truth_words)

    # harmonic mean of precision and recall
    f1 = 2 * precision * recall / (precision + recall) if (precision + recall) else 0

    return {
        "precision":          round(precision, 3),
        "recall":             round(recall, 3),
        "f1":                 round(f1, 3),
        "words_in_truth":     len(truth_words),
        "words_extracted":    len(extracted_words),
        "words_correct":      len(overlap),
        "words_missed":       len(missed := truth_words - extracted_words),   # in truth, not extracted
        "words_hallucinated": len(extracted_words - truth_words)              # extracted, not in truth
    }

# --- run for each model ---
